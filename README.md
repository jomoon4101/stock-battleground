# 주식 배틀그라운드

100명의 트레이더가 30개 익명 종목으로 40턴 동안 경쟁하는 온라인 브라우저 게임 프로토타입입니다. 실제 사용자가 방 코드로 참가하고, 방장이 시작할 때 남은 자리는 AI 봇이 채웁니다.

## 실행

### 가장 쉬운 방법—Windows

`게임실행.cmd`를 더블클릭하면 서버가 시작되고 게임 페이지가 자동으로 열립니다.

`index.html`을 직접 더블클릭하는 방식으로는 온라인 게임이 실행되지 않습니다. 이 게임은 턴, 거래와 방 참가를 처리하는 Node.js 서버가 반드시 필요하며, 브라우저의 `file://` 보안 정책상 모듈도 직접 로드할 수 없습니다. 홈 화면만 보인다면 반드시 `게임실행.cmd` 또는 아래 서버 명령으로 실행하세요.

### 명령으로 실행

Node.js 18 이상에서 다음 명령을 실행합니다.

```powershell
npm run dev
```

이후 브라우저에서 `http://127.0.0.1:4173`을 엽니다. 별도 패키지 설치나 빌드는 필요하지 않습니다.

1. 첫 사용자가 `온라인 방 만들기`를 누릅니다.
2. 대기실의 6자리 방 코드를 다른 사용자에게 전달합니다.
3. 다른 사용자는 닉네임과 방 코드를 입력하고 `방 참가`를 누릅니다.
4. 방장이 시작하면 빈자리가 AI로 채워지고 서버가 게임을 진행합니다.

같은 와이파이의 다른 기기는 서버 PC의 내부 IP와 포트로 접속할 수 있습니다. 인터넷에서 공개하려면 아래 Docker 이미지를 HTTPS를 지원하는 서버에 배포해야 합니다.

### 다른 사람과 테스트

- 같은 PC: 일반 창에서 방을 만들고 시크릿 창에서 같은 주소를 연 뒤 방 코드로 참가합니다.
- 같은 와이파이: `게임실행.cmd` 실행 시 표시되는 `http://내부IP:4173` 주소를 전달합니다. Windows 방화벽이 물으면 Node.js의 개인 네트워크 접근을 허용해야 합니다.
- 인터넷에 있는 친구: Docker 설정을 클라우드 서버에 배포하고 발급된 HTTPS 주소를 공유해야 합니다. 클라우드 계정과 배포 대상이 있어야 하므로 이 프로젝트만으로 임의의 공개 주소를 만들 수는 없습니다.

```powershell
docker build -t stock-battleground .
docker run --rm -p 4173:4173 stock-battleground
```

테스트:

```powershell
npm test
```

프로덕션 프런트 빌드와 로컬 미리보기:

```powershell
npm run build
npm run preview
```

`npm run build`는 배포 산출물을 `dist`에 생성합니다. `npm run preview`는 해당 `dist`와 게임 API를 `http://127.0.0.1:4174`에서 함께 실행합니다. `npm start`는 일반 게임 서버 실행 명령입니다.

## 구현된 기능

- 100명 참가, 빈자리 99개 AI 봇 자동 충원
- 방 생성·6자리 코드 참가·방장 시작·새로고침 재접속
- 서버 전송 이벤트 기반 실시간 상태 동기화
- 가격·턴·거래·아이템을 서버에서 판정하는 권한형 구조
- 미래 가격과 다른 사용자의 포트폴리오 비공개
- 40턴, 터보 약 18분·빠른 게임 45분·공식 게임 100분 시간표
- 5개 시장의 익명 종목 30개와 매 게임 새로 생성되는 가격 패턴
- 선차트·봉차트 전환, OHLC 데이터와 거래량 차트
- 종목명·현재가·변동률 오름차순/내림차순 정렬
- 즉시 매수·매도, 다음 턴부터 작동하는 지정가 예약 주문
- 실시간 순자산 순위와 10·20·30·40턴 순위 잠금
- 순위 클릭 시 상대의 최대 보유 종목 표시
- 내 자산 클릭 시 보유 종목 목록과 종목별 빠른 매수·매도 이동
- 순위 상승·하락 애니메이션과 연속 자산 상승 강조
- 10·20·30·40턴 종료 10초 전 상단바 카운트다운
- 시작 및 매 10턴 종료 시 월급 지급과 자산 구간별 누진 세금
- 월급 10배 한도 대출, 선이자·정기이자·상환
- 10턴 만기 채권
- 특별 아이템 7종, 랜덤 아이템 2종
- 99개 봇의 매수·매도·대출·채권 의사결정
- 최종 순위와 상위 3명 결과 화면

## 온라인 MVP의 운영 범위

- 방과 세션은 현재 서버 메모리에 저장됩니다. 서버가 재시작되면 진행 중인 방이 사라집니다.
- 한 서버 인스턴스에서 여러 방을 운영할 수 있지만, 여러 서버로 수평 확장하려면 Redis와 공유 데이터베이스가 필요합니다.
- 정식 서비스에는 계정 인증, 신고·제재, 접속 제한, 운영자 도구, 영구 전적 저장과 HTTPS 배포를 추가해야 합니다.
- `/api/health`에서 서버 상태를 확인할 수 있습니다.

## 환경변수

필요한 이름은 `.env.example`에 정리되어 있습니다. 실제 값이나 비밀 키는 포함하지 않습니다.

- `VITE_API_BASE_URL`: Vercel 프런트가 연결할 외부 게임 서버의 HTTPS 주소. 로컬 통합 실행에서는 비워둡니다.
- `PORT`: Node 게임 서버 포트.
- `ALLOWED_ORIGINS`: 게임 서버가 허용할 프런트 Origin. 여러 개면 쉼표로 구분합니다.

이 프로젝트는 현재 Supabase를 사용하지 않습니다. 따라서 `VITE_SUPABASE_URL`과 `VITE_SUPABASE_ANON_KEY`도 코드에서 사용하지 않으며 `.env.example`에 추가하지 않았습니다.

## Vercel 배포

현재 프로젝트는 Vite나 React Router 기반이 아닌 바닐라 ES Module 프런트입니다. 별도 의존성 없는 빌드 스크립트가 `dist`를 만들며, `vercel.json`이 빌드 명령과 출력 디렉터리를 지정합니다. React Router는 사용하지 않지만, 향후 클라이언트 경로가 생겨도 새로고침 404가 나지 않도록 정적 파일 우선 SPA fallback을 포함했습니다.

중요: Vercel에는 프런트만 배포합니다. 현재 게임 서버는 메모리 방 상태와 장시간 SSE 연결을 사용하므로 Vercel 서버리스 함수에 적합하지 않습니다. 게임 서버는 Dockerfile을 사용해 Railway, Render, Fly.io 또는 일반 VPS 같은 상태형 호스팅에 별도로 배포해야 합니다.

배포 순서:

1. Docker 게임 서버를 상태형 호스팅에 배포하고 HTTPS 주소를 발급받습니다.
2. 게임 서버의 `ALLOWED_ORIGINS`에 Vercel 도메인을 설정합니다.
3. Vercel 프로젝트 환경변수 `VITE_API_BASE_URL`에 게임 서버 HTTPS 주소를 설정합니다.
4. Vercel 프로젝트 루트를 이 폴더로 지정하고 배포합니다.
5. Build Command는 `npm run build`, Output Directory는 `dist`입니다. `vercel.json`에도 동일하게 설정되어 있습니다.

- [ ] Vercel의 `VITE_API_BASE_URL`이 실제 HTTPS 게임 서버 주소인지 확인한다.
- [ ] 게임 서버의 `ALLOWED_ORIGINS`에 정확한 Vercel Origin을 넣는다.
- [ ] 게임 서버 `/api/health`가 HTTPS로 응답한다.
- [ ] 프런트와 게임 서버 모두 HTTPS를 사용해 혼합 콘텐츠 오류를 방지한다.

## 프로토타입 규칙 보완

원안에서 수치가 없거나 해석이 여러 개인 부분은 아래처럼 정했습니다.

- 순자산은 `현금 + 주식 평가액 + 채권 원금 - 대출원금`입니다.
- 세율은 순자산 구간에 따라 5%, 10%, 15%, 20%, 30%입니다.
- 월급은 시작 직후와 10·20·30·40턴 종료 시 지급합니다.
- 채권은 최소 10만원, 10턴 만기, 수익률 5%입니다.
- 순자산이 음수이면 신규 대출과 아이템 구매가 막히고 월급 20% 감소, 정기 대출이자 5% 가산이 적용됩니다.
- 랜덤 포트폴리오 교체는 기존 주식 평가액을 보존해 2~5개 무작위 종목으로 재분배하고, 사용한 턴이 끝날 때까지만 수동 거래를 잠급니다.
- 실제 기업의 과거 원시 데이터 대신 국가별 변동성·추세·충격을 반영한 합성 역사 패턴을 씁니다. 실제 시세 데이터 연동은 서버와 데이터 라이선스가 필요한 다음 단계입니다.
